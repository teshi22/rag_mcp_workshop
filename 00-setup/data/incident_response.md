# クラウド障害対応手順

## 目的
Azure 上で障害が発生した際のエスカレーションと復旧手順を定めます。

## 障害レベル定義

| レベル | 定義 | 対応 SLA |
|--------|------|----------|
| P1（緊急） | 本番サービス全停止、データ損失の可能性 | 15 分以内に初動、4 時間以内に復旧 |
| P2（重大） | 本番サービスの一部機能が利用不可 | 30 分以内に初動、8 時間以内に復旧 |
| P3（軽微） | 性能劣化、非本番環境の障害 | 翌営業日中に対応 |
| P4（情報） | 監視アラート、計画メンテナンス | 週次で確認 |

## エスカレーションフロー

### P1・P2 の場合
1. **検知**: Azure Monitor アラートまたはユーザー報告
2. **初動** (15分以内):
   - 障害対応チャンネル `#incident-response` に報告
   - 担当チームの on-call エンジニアに連絡
3. **調査** (並行して実施):
   - Azure Service Health で Azure 側の障害確認
   - Application Insights でアプリログ確認
   - リソースのメトリクス（CPU、メモリ、レイテンシ）確認
4. **復旧対応**:
   - アプリ再起動: `az webapp restart` / `az functionapp restart`
   - スケールアウト: インスタンス数の一時的な増加
   - フェイルオーバー: DR リージョン（Japan West）への切り替え
5. **報告**: 復旧後 24 時間以内にポストモーテムを作成

### P3・P4 の場合
1. JIRA チケットを作成（ラベル: `cloud-incident`）
2. 担当チームのバックログに追加
3. 週次の振り返りで対応方針を決定

| 役割 | 連絡先 |
|------|--------|
| クラウド推進チームリーダー | cloud-team@example.com |
| on-call エンジニア | PagerDuty 経由 |
| セキュリティインシデント | security@example.com |
| Azure サポート（Premier） | Azure ポータルからチケット起票 |

## 定期訓練
- 四半期ごとに障害対応訓練（テーブルトップエクササイズ）を実施
- DR フェイルオーバーテストは半期に 1 回実施
